Understanding Ansible

Simple, Agentless IT Automation

What is Ansible?

Ansible is an open-source automation platform that simplifies IT operations by automating configuration management, application deployment, cloud provisioning, and many other IT processes. Created by Michael DeHaan in 2012 and later acquired by Red Hat in 2015, Ansible has become one of the most popular automation tools in the DevOps ecosystem.

Unlike traditional automation tools, Ansible uses a simple, human-readable language (YAML) and requires no agents to be installed on managed systems. It connects to your servers via SSH (for Linux/Unix) or WinRM (for Windows), making it lightweight, secure, and easy to get started.

Common Use Cases

Where Ansible Shines:

  • Server Configuration: Install packages, configure services, manage users and permissions
  • Application Deployment: Deploy web applications, databases, and microservices
  • Cloud Provisioning: Create and manage cloud resources on AWS, Azure, GCP
  • Security Automation: Apply security patches, enforce firewall rules, manage SSL certificates
  • Network Automation: Configure routers, switches, and load balancers
  • Disaster Recovery: Quickly rebuild infrastructure from code
  • Continuous Deployment: Integrate with CI/CD pipelines for automated releases

How Ansible Works

Ansible follows a simple push-based architecture:

  1. You write playbooks in YAML describing the desired state of your systems
  2. Ansible connects to your servers via SSH (no agents required)
  3. Tasks are executed in order on the target machines
  4. Results are reported back, showing what changed and what didn't

Idempotency: Running the same playbook multiple times produces the same result. If a server is already in the desired state, Ansible won't make unnecessary changes.

Ansible Directory Structure

Ansible organizes automation code using a standardized directory structure. Here's a typical setup:

webserver/
└── tasks/
    ├── main.yml       # Entry point - includes other task files
    ├── apt.yml        # Package installation tasks
    ├── service.yml    # Service management tasks
    └── write.yml      # File writing and configuration tasks

Understanding the Structure:

  • tasks/main.yml: The main entry point that orchestrates which tasks run and in what order
  • tasks/apt.yml: Contains all package installation and update tasks (uses apt, yum, or other package managers)
  • tasks/service.yml: Manages system services - starting, stopping, enabling, or disabling them
  • tasks/write.yml: Handles file operations - creating files, copying content, setting permissions
Why Organize This Way?

Separating tasks into different files makes your automation more maintainable and reusable. You can:

  • Run only package updates without touching services
  • Test file configurations independently
  • Reuse task files across different roles
  • Understand what each file does at a glance

Task Execution Flow

Typical execution sequence:

1. main.yml → includes → 2. apt.yml (install packages) → 3. service.yml (start services) → 4. write.yml (configure files)

This logical flow ensures that:

  • Packages are installed before you try to start services
  • Services exist before you configure them
  • Configuration files are written after all dependencies are in place

Getting Started

To start using Ansible:

  1. Install Ansible on your control machine (your laptop or a dedicated server)
  2. Create an inventory file listing your target servers
  3. Write a simple playbook describing what you want to automate
  4. Run the playbook with the ansible-playbook command

No installation needed on target servers! As long as they have SSH and Python, Ansible can manage them.
